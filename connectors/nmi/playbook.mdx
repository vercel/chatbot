# NMI Connector Playbook (NewLeaf)

## Architecture
NMI is wrapped **INSIDE Hyperswitch** as a connector. We send tokens + transactions through Hyperswitch, which uses the NMI security key to call Allied Payments (`ap.transactiongateway.com`).

## CIT vs MIT (MUST UNDERSTAND)

### CIT (Customer-Initiated Transaction)
- First charge, card-on-file setup
- CVV **required** (captured via Collect.js iframe)
- Customer IP **must** be captured
- Field `initiated_by`: `"customer"`
- Field `stored_credential_indicator`: `"stored"` (first setup)
- Day 0 authorization authenticates the card for future MIT charges

### MIT (Merchant-Initiated Transaction)
- Recurring subscription charges
- No CVV (stored on vault, not passed)
- No customer presence (no IP should be sent)
- Field `initiated_by`: `"merchant"`
- Field `stored_credential_indicator`: `"used"`
- Field `billing_method`: `"recurring"`
- Field `initial_transaction_id`: Day 0 CIT txn ID (consent proof)

## Golden Vault Flow
1. Customer fills card via Collect.js iframe (PCI isolated)
2. We do $0 auth (CIT) with customer IP + `customer_acceptance` â†’ captures `network_transaction_id` as `nmiDayZeroTransactionId`
3. Create vault from token â†’ stores `customer_vault_id` as `nmiVaultId`
4. Create subscription pointing at vault
5. Every recurring charge passes `initial_transaction_id=<Day 0 txn>`

## Hard Decline Codes (DO NOT retry)
- **250**: Declined (generic)
- **251**: Insufficient funds (smart retry may apply â€” once only)
- **252**: Invalid card number
- **253**: Expired card
- **254**: CVV mismatch (CIT only)
- Retrying 250-254 more than once triggers **fraud velocity flags** at Allied Payments
